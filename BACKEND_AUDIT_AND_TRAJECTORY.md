# Аудит `diplom-backend` и траектория завершения проекта

## 1) Краткий вердикт

Проект действительно выглядит как **backend-only** (признаков React/Frontend-кода внутри репозитория не обнаружено), но текущая реализация находится на стадии прототипа и пока не готова как API для production/дипломной защиты.

Текущее состояние можно охарактеризовать как:
- Есть заготовка доменной модели (users/tasks/tags/events).
- Есть базовая структура Django-приложений и роутинг.
- Бизнес-логика, API-контракты, тесты, безопасность и инфраструктура почти не доведены до рабочего состояния.

## 2) Проверка: есть ли в репозитории фронтенд

Что проверено:
- Структура папок и файлов в `diplom-backend`.
- Поиск типичных frontend-артефактов (`package.json`, `react`, `vite`, `webpack`, `npm`, `yarn`).

Результат:
- Явных frontend-файлов в `diplom-backend` нет.
- Репозиторий соответствует идее backend-проекта.

## 3) Что уже сделано хорошо

- Создан Django-проект и несколько приложений: `accounts`, `tasks`, `tags`, `events`.
- Настроен кастомный пользователь (`AUTH_USER_MODEL`).
- Подключен PostgreSQL в `settings.py`.
- Добавлен `corsheaders` (есть попытка обеспечить интеграцию с React-клиентом).
- Часть моделей и миграций уже создана.

## 4) Критические проблемы (блокеры завершения)

## Архитектурные и структурные
- В проекте одновременно присутствует старое приложение `general` (с дублированием моделей) и новые приложения `accounts/tasks/tags/events`. Это признак незавершенного рефакторинга.
- `general` не подключен в `INSTALLED_APPS`, но его миграции и модели остались в репозитории, что создает путаницу.
- В `accounts/views.py` импортируются модели из `accounts.models` как `Task, Tag, Event`, хотя их там нет.

Приложение `general` удален, поправлены views.

## API и бизнес-логика
- В class-based views методы объявлены с неправильной сигнатурой (`def get(request)` вместо `def get(self, request, ...)`).
- Вьюхи используют `GET` там, где должны быть `POST/PUT/PATCH/DELETE`.
- `TaskList` пытается читать `request.body` в `GET` и вызывает `Task.objects.all(username)`, что некорректно.
- `accounts/urls.py` фактически пустой (роуты закомментированы), но auth-флоу частично реализован.
- В `project/urls.py` отсутствает подключение `accounts.urls`, то есть регистрация/логин не прокинуты в основной роутинг.

В `project/urls.py` добавлен `accounts.urls`, роуты в `accounts/urls.py` разкомментированы.
Над корректными views ведется работа.

## Модель данных
- В `CustomUser` есть поле `password_hash` (у `AbstractUser` уже есть поле `password`) и оно ограничено `max_length=15`, что нефункционально и потенциально ломает модель.
- В `TypeField.__str__` возвращается `self.name`, но поля `name` нет (будет ошибка в runtime).
- В `tasks.models.Task` поле `tag` одиночное (`ForeignKey`), хотя по смыслу задачи обычно имеют несколько тегов (`ManyToMany`).
- Названия полей типа `user_id`, `task_id` для `ForeignKey` не критичны, но не соответствуют стандартному стилю Django (лучше `user`, `task`).

Поле `password_hash` убрано, `TypeField.__str__` теперь возвращает корректные поля, названия для полей приведенны к стандароному стилю Djdngo.
Над связью `ManyToMany` ведется работа.


## Безопасность и конфигурация
- `SECRET_KEY` захардкожен.
- `DEBUG=True` и `ALLOWED_HOSTS=["*"]`.
- CORS-конфиг избыточен и устаревший (`CORS_ORIGIN_WHITELIST` дублируется с `CORS_ALLOWED_ORIGINS`).
- Используется `@csrf_exempt` для auth-вьюх.

Ведется работа.

## Качество и поддерживаемость
- Практически отсутствуют тесты (файлы `tests.py` пустые).
- Нет `README`, нет `requirements.txt`/`pyproject.toml`.
- Непоследовательность версий: миграции встречаются от Django 4.2 и 6.0.

Ведется работа.

## 5) Оценка готовности (реалистично)

- **Техническая готовность backend к интеграции с фронтом**: ~20-30%.
- **Готовность к демонстрации как дипломный MVP**: ~35% (с учетом задела по моделям).
- **Готовность к production**: <10%.

## 6) Траектория завершения проекта

Ниже план, который приведет проект к состоянию «рабочий backend для React task manager».

### Этап 0. Стабилизация репозитория (1-2 дня)

Цель: убрать хаос и получить запускаемый baseline.

Задачи:
- Зафиксировать единую архитектуру приложений: оставить `accounts/tasks/tags/events`, убрать или архивировать `general`.
- Привести миграции в консистентное состояние (либо пересобрать с нуля, либо аккуратно выстроить цепочку).
- Добавить `requirements.txt` (или `pyproject.toml`) и базовый `README` с командами запуска.
- Вынести секреты и параметры БД в `.env`.

Критерий готовности:
- Проект поднимается командой `python manage.py runserver`.
- `python manage.py makemigrations && python manage.py migrate` выполняются без ошибок.

### Этап 1. Базовый API-контур (3-5 дней)

Цель: закрыть минимально необходимый функционал для фронта.

Задачи:
- Подключить DRF (`djangorestframework`), сделать serializers.
- Внедрить JWT-аутентификацию (например, `djangorestframework-simplejwt`).
- Реализовать API:
  - `POST /api/auth/register`
  - `POST /api/auth/login`
  - `POST /api/auth/refresh`
  - `GET/POST /api/tasks`
  - `GET/PATCH/DELETE /api/tasks/{id}`
  - `GET/POST /api/tags`
  - `GET/POST /api/events`
- Добавить фильтрацию задач (по статусу, дедлайну, тегам) и пагинацию.
- Удалить `csrf_exempt` из API и перейти на token-based auth.

Критерий готовности:
- Frontend может зарегистрировать пользователя, залогиниться и выполнять CRUD задач.

### Этап 2. Доработка доменной логики (3-4 дня)

Цель: сделать продукт «крутым таск-менеджером», а не просто CRUD.

Задачи:
- Добавить статусы задач (`todo`, `in_progress`, `done`).
- Добавить приоритеты и сортировку.
- Сделать повторяемые задачи (ежедневно/еженедельно/ежемесячно) с понятной схемой.
- Перевести `Task.tag` на `ManyToMany` (или через промежуточную таблицу), если нужны множественные теги.
- Определить назначение `ExtraTaskField`; либо довести до use-case, либо убрать как преждевременную сложность.

Критерий готовности:
- Есть осмысленные пользовательские сценарии, а не только «сохранить/удалить запись».

### Этап 3. Качество, тесты, наблюдаемость (3-5 дней)

Цель: защитить проект от регрессий и подготовить к защите.

Задачи:
- Покрыть тестами критический контур:
  - auth flow;
  - права доступа (пользователь видит только свои задачи);
  - CRUD задач/тегов;
  - валидация входных данных.
- Добавить линтер/форматтер (`ruff`/`flake8`, `black`, `isort`) и pre-commit.
- Нормализовать обработку ошибок и структуру API-ответов.
- Добавить логирование ключевых событий.

Критерий готовности:
- Тесты стабильно проходят в CI, код единообразен.

### Этап 4. Подготовка к релизу и защите (2-3 дня)

Цель: «продаваемый» результат для диплома.

Задачи:
- Подготовить OpenAPI/Swagger-документацию.
- Написать полноценный `README`:
  - архитектура;
  - запуск локально;
  - переменные окружения;
  - примеры API-запросов.
- Настроить Docker Compose (backend + postgres), при необходимости `nginx`.
- Подготовить демо-сценарий для защиты: 3-5 пользовательских флоу.

Критерий готовности:
- Любой проверяющий запускает проект по инструкции за 5-10 минут.

## 7) Приоритеты на ближайшую неделю (практический backlog)

`P0` (сразу):
- Удалить/заморозить `general`.
- Починить модели и вьюхи до рабочего состояния.
- Включить `accounts`-роуты и сделать рабочий auth API.

`P1`:
- Перейти на DRF + JWT.
- Закрыть CRUD задач/тегов/событий с правами доступа.

`P2`:
- Написать тесты на critical path.
- Документация + Docker + CI.

## 8) Итоговая рекомендация как ментору

Студент сделал хороший каркас и предметную декомпозицию, но проект пока на уровне «черновой backend-заготовки».  
Для доведения до сильной дипломной работы нужен фокус не на добавлении новых сущностей, а на:
- консистентности архитектуры,
- корректном REST API,
- безопасности,
- тестируемости и воспроизводимом запуске.

Если идти по траектории выше, проект можно довести до уверенного MVP примерно за **2-3 недели** плотной работы.
